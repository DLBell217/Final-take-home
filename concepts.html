21. HTTP overview

  1. HTTP Protocols: HTTP (Hypertext Transfer Protocol) is the foundation of data communication on the World Wide Web.
  2. HTTP Methods: HTTP methods specify the type of action to be performed on a resource. Common methods include GET, POST, PUT, DELETE, etc.
  3. Request Message: A request message is sent by a client to a server to request a specific action, typically including a URL, HTTP method, headers, and sometimes a body containing data.
  4. Response Message: A response message is sent by a server to a client in response to a request, typically containing a status code, headers, and sometimes a body containing data.
  5. Message Header Fields: Message header fields contain additional information about the request or response, including metadata such as content type, content length, cookies, caching directives, and more.
  6. Response Status Codes: Response status codes indicate the success or failure of a request. They are three-digit numbers categorized into different classes, such as 1xx for informational, 2xx for success, 3xx for redirection, 4xx for client errors, and 5xx for server errors. Some common status codes include 200 OK, 404 Not Found, 500 Internal Server Error, etc.

22. Restful Server

  1. RESTful Services: REST (Representational State Transfer) is an architectural style for designing networked applications, and RESTful services are web services that adhere to the principles of REST, such as statelessness, uniform interface, and client-server separation.
  2. RESTful API Endpoints: RESTful API endpoints are URLs (Uniform Resource Locators) that represent resources on the server and provide access to data or operations through HTTP methods, such as GET, POST, PUT, DELETE.
  3. Routing: Routing is the process of determining how an application responds to client requests at specific endpoints. It involves mapping URLs to server-side functions or handlers that process the request and generate a response.
  4. RESTful API: A RESTful API is an API (Application Programming Interface) that follows the principles of REST, providing access to resources over HTTP using standard methods and formats, such as JSON or XML.
  5. Client: A client is a software application or system that makes requests to a server over a network, typically using HTTP, to access resources or services provided by the server.
  6. Clean URL: Clean URLs, also known as pretty URLs or search engine-friendly URLs, are URLs that are easy to read and understand, typically containing meaningful words or phrases that describe the content of the resource.
  7. Resource: In the context of RESTful services, a resource is a data entity or object that can be accessed or manipulated through the API, such as a user, product, or post.
  8. Endpoints: Endpoints are specific URLs within a RESTful API that represent resources and define the operations that can be performed on those resources using HTTP methods, such as GET, POST, PUT, DELETE. Each endpoint typically corresponds to a specific resource or collection of resources.

23. HTTP methods

  1. POST: Used to submit data to be processed to a specified resource. The data is included in the body of the request, typically used for creating new resources on the server.
  2. GET: Requests data from a specified resource. The data is specified in the URL of the request, typically used for retrieving information from the server.
  3. PUT: Replaces all current representations of the target resource with the request payload, typically used for updating existing resources on the server.
  4. PATCH: Partially updates a resource on the server with the request payload, typically used for modifying existing resources with partial data.
  5. DELETE: Deletes the specified resource from the server, typically used for removing resources from the server.
  6. OPTIONS: Returns the HTTP methods that the server supports for the specified URL, typically used for discovering the capabilities of the server.
  7. HEAD: Similar to GET, but returns only the response headers without the response body, typically used for retrieving metadata about a resource without transferring the entire content.
  8. TRACE: Echoes the received request so that a client can see what (if any) changes or additions have been made by intermediate servers, typically used for diagnostic purposes.
  9. CONNECT: Establishes a tunnel to the server identified by the target resource, typically used for establishing a secure communication channel through an HTTP proxy.

25. Cross-origin resource sharing

The same-origin policy is a security measure implemented by web browsers that restricts interactions between resources (such as JavaScript, CSS, and HTML) from different origins (domains, protocols, and ports). It prevents a web page or script from making requests to another domain unless they share the same origin, protecting users from cross-site scripting (XSS) attacks and other security vulnerabilities.

26. three phases

when a component is being initialized and inserted into the DOM. It involves the creation of the component instance, setting initial state and props, and rendering the component for the first time.

when a component's state or props change. It involves re-rendering the component with updated data and potentially performing side effects such as fetching data or updating the DOM.

when a component is being removed from the DOM. It involves cleaning up any resources or event listeners associated with the component to prevent memory leaks or performance issues.

27. React Hooks

used to add state to functional components in React. It allows you to declare state variables and update them, triggering re-renders when the state changes.

used to perform side effects in functional components. It allows you to execute code in response to component updates, such as fetching data from an API, subscribing to events, or updating the DOM.

used to access the context in functional components. It allows you to consume context values provided by a Context Provider higher up in the component tree without needing to pass props down manually.

used to create mutable references to DOM elements or other values that persist across renders. It allows you to access and modify the current value of a reference without causing re-renders.

used to manage complex state logic in functional components. It allows you to dispatch actions to update state based on the previous state, similar to how reducers work in Redux.

used to memorize callback functions in functional components. It allows you to prevent unnecessary re-renders by memorizing the callback function when its dependencies haven't changed.

used to memorize values in functional components. It allows you to memorize the result of expensive computations so that they are only recalculated when their dependencies change, improving performance.
